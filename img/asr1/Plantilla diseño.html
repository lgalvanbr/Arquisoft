Plantilla de diseño del experimento — ASR1 (latencia y bloqueo)

Título del experimento Validación de latencia end-to-end (< 2 s) para detección de intentos fallidos y restricción temporal (IntentoBloqueado) con ALB y dos instancias EC2
ASRs involucrados ASR1 — Latencia: como consultor tecnológico de BITE.CO, ante múltiples intentos fallidos de autenticación (con el sistema operando correctamente), se debe detectar el comportamiento anómalo y restringir el acceso temporalmente en menos de 2 segundos.
Propósito del experimento Comprobar si el diseño desplegado (ALB + instancia + modelo de bloqueo) satisface el ASR1 del sprint: tiempo total desde la petición en el balanceador hasta la persistencia del bloqueo inferior a 2 s, con health checks estables y registro de auditoría.
Resultados esperados Health Check del balanceador en estado Healthy bajo carga JMeter; registro en AuditoriaAcceso y, ante reincidencia, creación de bloqueo en IntentoBloqueado con activo = True y fecha_bloqueo_hasta; latencia medida en JMeter < 2000 ms end-to-end; uso correcto de la IP del cliente vía balanceador para esta_bloqueado(); consumo de recursos de la instancia dentro de límites operativos sin fallos del ALB.
Elementos de arquitectura Application Load Balancer report-alb orientado a Internet; dos instancias EC2 t2.nano como backends (report-app-lb-a, report-app-lb-b); Security Groups (report-trafico-lb, report-trafico-http, report-trafico-ssh, report-trafico-db); Target Group report-app-group con health checks en HTTP:8080; base de datos en instancia dedicada y modelos Django (IntentoBloqueado, AuditoriaAcceso); cliente de prueba JMeter contra el endpoint de login del balanceador.
Estilos de arquitectura asociados y argumentación Microservicios: autenticación, usuarios, recursos cloud, reportes y empresa como servicios independientes para desplegar y escalar por separado, mejorar mantenibilidad y evolución del sistema. En el experimento se valida el flujo crítico de autenticación/bloqueo expuesto tras el ALB.
Tácticas asociadas y argumentación Distribución de carga: el balanceador concentra el tráfico y prepara el escalado horizontal si se añaden instancias. Aislamiento de recursos: separación de bases de datos de autenticación y de negocio para reducir contención y mejorar desempeño.
Infraestructura computacional requerida ALB público (report-alb, Internet-facing, IPv4); 2× EC2 t2.nano (report-app-lb-a y report-app-lb-b) en us-east-1; 1× EC2 t3.micro para base de datos (report-db); reglas de Security Group (HTTP 8080 desde el ALB hacia las instancias); Target Group report-app-group con 2 targets healthy y health check activo; estación de pruebas con JMeter; conectividad a la base de datos del servicio de autenticación.
Tecnologías y argumentación AWS ALB, EC2, Security Groups y Target Groups; aplicación (p. ej. Django) con modelos IntentoBloqueado y AuditoriaAcceso; Apache JMeter para hilos concurrentes y medición de latencia en el endpoint de login expuesto por el DNS del balanceador.
Descripción del experimento Se envían ráfagas de solicitudes de login fallidas al ALB; el tráfico llega a la instancia activa; el servidor aplica la lógica de detección y persiste en intentos_bloqueados con activo y fecha_bloqueo_hasta. Las métricas se obtienen de los reportes de JMeter (tiempo desde envío hasta respuesta de bloqueo) y de la consola AWS (estado del target y del balanceador). Se contrasta el tiempo total con el umbral de 2 s del ASR1.
Plan de uso de IAG Se utilizó IAG (GitHub Copilot) para: (1) generación de la estructura base del plan de pruebas JMeter, (2) revisión de configuración de Security Groups en AWS, y (3) redacción y estructuración del documento de evidencia del experimento.
